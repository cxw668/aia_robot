## 一、项目概览

AIA Robot 是面向友邦保险（AIA）用户的 RAG（检索增强生成）智能客服智能体。用户输入保险相关问题后，系统通过向量检索从本地知识库召回最相关的内容，结合 LLM 生成准确、自然的中文回答。

### 核心目标


| 目标       | 说明                                                                      |
| -------- | ----------------------------------------------------------------------- |
| **低延迟**  | 将远程 Qdrant 云服务迁移至本地 Docker，消除网络 RTT，检索响应 < 200ms                        |
| **高精度**  | 基于 `BAAI/bge-small-zh-v1.5` embedding，结合跨 collection 回退策略，提升 Top-K 召回质量 |
| **数据完整** | 覆盖 `aia_data` 目录全量数据：服务分类、产品信息、表单、分公司页面、反欺诈提示                           |
| **可扩展**  | 模块化架构，后续可接入 PDF 文档、多模型、多语言                                              |

## 三、技术架构

```
用户输入
   |
   v
[FastAPI 后端]  路由: POST /chat
   |
   |---> [Redis 缓存层]  命中则直接返回
   |
   v
[RAG 检索引擎 - app/knowledge_base/rag.py]
   |
   |---> [SentenceTransformer]  BAAI/bge-small-zh-v1.5
   |       encode(query) -> 512 维向量
   |
   |---> [本地 Qdrant - Docker :6333]
   |       query_points() -> Top-K 相关文档
   |
   |---> 跨 collection 回退策略
   |       优先检索匹配 collection
   |       含《》符号 -> 优先检索表单类 collection
   |       目标不存在则全库检索并按 score 合并
   |
   v
[LLM - SiliconFlow API]
   |  模型: tencent/Hunyuan-MT-7B
   |  prompt = 系统指令 + 知识库上下文 + 用户问题
   |
   v
[流式响应] -> 前端 React 展示
```

### 基础设施组件


| 组件           | 技术选型                        | 用途                  |
| ------------ | --------------------------- | ------------------- |
| 向量数据库        | Qdrant (Docker 本地)          | 存储 embedding，执行语义检索 |
| Embedding 模型 | `BAAI/bge-small-zh-v1.5`    | 中文语义向量化（512 维）      |
| LLM          | SiliconFlow / Hunyuan-MT-7B | 自然语言生成              |
| 关系数据库        | MySQL                       | 用户、会话、历史记录          |
| 缓存           | Redis                       | 高频问答缓存，降低 LLM 调用成本  |
| 对象存储         | MinIO                       | PDF 原文件及解析结果存储      |
| 后端框架         | FastAPI + Uvicorn           | 异步 API 服务           |
| 前端框架         | React + Vite + TypeScript   | 用户交互界面              |

### Embedding 参数


| 参数         | 值                                                      | 说明                          |
| ---------- | ------------------------------------------------------ | --------------------------- |
| 模型         | `BAAI/bge-small-zh-v1.5`                               | 轻量中文模型，推理快                  |
| 向量维度       | 512                                                    | 固定输出维度                      |
| 相似度类型      | Cosine                                                 | `normalize_embeddings=True` |
| JSON 分块    | 按条目独立                                                  | 每条 JSON 记录作为一个向量点           |
| 文本分块大小     | 600 字符                                                 | 适用于长文本（txt、PDF）             |
| 分块重叠       | 80 字符                                                  | 保证上下文连续性                    |
| Payload 字段 | title, content, service_name, service_url, source_file | 检索后用于展示和溯源                  |

## 五、构建阶段路线图

### Phase 0：本地 Qdrant 环境准备（当前优先执行）

**目标**：用 Docker 在本地启动 Qdrant，替换远程云服务，消除网络延迟。


| 步骤  | 操作                       | 验证方式                                  |
| --- | ------------------------ | ------------------------------------- |
| 0-1 | 确认 Docker Desktop 已安装并运行 | `docker info` 无报错                     |
| 0-2 | 拉取 Qdrant 官方镜像           | `docker pull qdrant/qdrant`           |
| 0-3 | 启动容器，挂载持久化存储卷            | 见下方命令                                 |
| 0-4 | 访问 Dashboard 验证服务        | `http://localhost:6333/dashboard` 可访问 |
| 0-5 | 更新 `.env` 中 Qdrant 配置    | 见下方配置                                 |
| 0-6 | 重启后端，验证连接正常              | 后端日志无 Qdrant 报错                       |


**Docker 启动命令**：

```powershell
docker run -d `
  --name qdrant-local `
  -p 6333:6333 `
  -p 6334:6334 `
  -v qdrant_storage:/qdrant/storage `
  qdrant/qdrant
```

`**.env` 更新内容**：

```env
QDRANTCLIENT_URL=http://localhost:6333
QDRANTCLIENT_KEY=
```

**验收标准**：`GET http://localhost:6333/collections` 返回 `{"result":{"collections":[]}}` 且 HTTP 200。

### Phase 2：RAG 检索质量优化

**目标**：在本地低延迟基础上，进一步提升检索准确性与覆盖率。


| 步骤  | 操作                                      | 预期收益                      |
| --- | --------------------------------------- | ------------------------- |
| 2-1 | 调整 `TOP_K`（当前为 3），评估 TOP_K=5 时的准确率与响应时间 | 更多候选提升覆盖率                 |
| 2-2 | 评估 score 阈值过滤（低于 0.5 的结果不注入 prompt）     | 减少噪声                      |
| 2-3 | 优化跨 collection 搜索合并策略（当前按 score 排序取前 K） | 防止单一 collection 占据所有 slot |
| 2-4 | 针对产品查询增加结构化过滤（按 productStatus=在售）       | 避免推荐停售产品                  |
| 2-5 | 测试同义词/口语化查询（如"退保手续"vs"退保流程"）            | 验证语义鲁棒性                   |


**验收标准**：对 20 条覆盖各 collection 的典型问题，Top-1 准确率 ≥ 85%。

### Phase 3：对话能力与多轮上下文

**目标**：让智能体支持多轮对话，保持上下文连贯性，提升用户体验。


| 步骤  | 操作                            | 说明                          |
| --- | ----------------------------- | --------------------------- |
| 3-1 | 审查 `app/session.py` 会话管理逻辑    | 确认历史消息存储格式与 TTL 设置          |
| 3-2 | 审查 `app/routers/chat.py` 对话接口 | 确认多轮历史如何拼入 prompt           |
| 3-3 | 设计系统 prompt 模板                | 角色定义 + 知识库上下文 + 历史对话 + 当前问题 |
| 3-4 | 评估历史轮次窗口（建议保留最近 5 轮）          | 防止 token 超限                 |
| 3-5 | 测试多轮对话场景                      | 见下方测试用例                     |


**多轮对话测试用例**：

```
用户: 我想退保，流程是什么？
机器人: [回答退保流程]
用户: 需要准备哪些材料？        <- 上文指代退保，应正确理解
机器人: [回答退保所需材料]
用户: 在上海能办理吗？           <- 上文指代退保+上海，应关联分公司信息
机器人: [回答上海分公司可办理退保]
```

**验收标准**：多轮指代消解正确率 ≥ 90%（基于 10 组多轮测试对话）。

### Phase 4：性能基准测试与调优

**目标**：量化系统各环节延迟，确保端到端响应满足用户体验要求。


| 环节           | 目标延迟    | 测量方式              |
| ------------ | ------- | ----------------- |
| Embedding 编码 | < 50ms  | 单次 encode() 计时    |
| Qdrant 检索    | < 100ms | query_points() 计时 |
| Redis 缓存命中   | < 5ms   | GET 操作计时          |
| LLM 首 token  | < 2s    | 流式接口首响应计时         |
| 端到端（无缓存）     | < 5s    | 从请求到完整回答          |
| 端到端（有缓存）     | < 200ms | 命中缓存后直接返回         |



| 步骤  | 操作                                   |
| --- | ------------------------------------ |
| 4-1 | 在 RAG 各环节加入计时日志（INFO 级别）             |
| 4-2 | 对 Qdrant 检索进行压测（并发 10 请求）            |
| 4-3 | 评估 Redis 缓存命中率（key 策略：query 文本 hash） |
| 4-4 | 如 Embedding 成为瓶颈，评估批量编码或模型量化方案       |
| 4-5 | 如 LLM 延迟过高，评估更换更快模型或启用流式输出           |


**验收标准**：本地 Qdrant 检索 P99 < 150ms；端到端无缓存 P50 < 4s。
