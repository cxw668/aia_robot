# AIA Robot 智能体构建计划

---

## 一、项目概览

AIA Robot 是面向友邦保险（AIA）用户的 RAG（检索增强生成）智能客服智能体。用户输入保险相关问题后，系统通过向量检索从本地知识库召回最相关内容，并结合 LLM 生成中文回答。

### 核心目标


| 目标         | 说明                                               |
| ------------ | -------------------------------------------------- |
| **低延迟**   | 使用本地 Docker Qdrant，降低网络 RTT，提升检索速度 |
| **高精度**   | 基于`BAAI/bge-small-zh-v1.5` 做中文语义向量检索    |
| **数据完整** | 覆盖`aia_data/` 当前全部核心数据文件               |
| **可扩展**   | 支持后续接入 PDF 文档、更多模型与多轮对话优化      |

## 三、技术架构

```text
用户输入
   |
   v
[FastAPI 后端]  POST /chat
   |
   |---> [Redis 缓存层] 命中则直接返回
   |
   v
[RAG 检索引擎 - app/knowledge_base/rag.py]
   |
   |---> [SentenceTransformer] BAAI/bge-small-zh-v1.5
   |       encode(query) -> 512 维向量
   |
   |---> [本地 Qdrant - Docker :6333]
   |       在 `aia_knowledge_base` 中检索 Top-K
   |
   v
[LLM - SiliconFlow API]
   |
   v
[流式响应] -> 前端 React 展示
```

### 基础设施组件


| 组件           | 技术选型                    | 用途                         |
| -------------- | --------------------------- | ---------------------------- |
| 向量数据库     | Qdrant（Docker 本地）       | 存储 embedding，执行语义检索 |
| Embedding 模型 | `BAAI/bge-small-zh-v1.5`    | 中文语义向量化（512 维）     |
| LLM            | SiliconFlow / Hunyuan-MT-7B | 自然语言生成                 |
| 关系数据库     | MySQL                       | 用户、会话、历史记录         |
| 缓存           | Redis                       | 高频问答缓存                 |
| 对象存储       | MinIO                       | PDF 原文件与解析结果         |
| 后端框架       | FastAPI + Uvicorn           | 异步 API 服务                |
| 前端框架       | React + Vite + TypeScript   | 用户交互界面                 |

## 五、Qdrant 设计

> 现阶段不是“多 collection 拆分”，而是**统一写入单集合**。

### 5.1 集合策略

- 集合名：`aia_knowledge_base`
- 写入脚本：`scripts/run_ingest_all.py`
- 核心方式：不同来源通过 payload 字段（如 `schema`、`source_file`、`service_name`）区分，检索时再做过滤/排序。

### 5.2 Embedding 参数


| 参数         | 值                       | 说明                  |
| ------------ | ------------------------ | --------------------- |
| 模型         | `BAAI/bge-small-zh-v1.5` | 中文检索主模型        |
| 向量维度     | 512                      | 固定输出维度          |
| 相似度类型   | Cosine                   | 归一化后余弦相似度    |
| 文本分块大小 | 600 字符                 | 长文本（txt/PDF）分块 |
| 分块重叠     | 80 字符                  | 保持上下文连续性      |

## 六、业务数据库

> 目标：避免与向量库重复存储知识正文，业务库仅存“元数据 + 业务状态 + 可追踪信息”。

### 6.1 设计原则

- 向量库（Qdrant）负责：知识正文 chunk、向量、语义检索。
- 业务库（MySQL）负责：知识入库记录、用户与会话、日志与追踪。
- 不在 MySQL 重复存储 `aia_data` 的完整正文内容。

### 6.2 业务数据库三类数据

#### 1）知识库（元数据）

建议存储：

- 数据源清单：`source_file`、`schema`、`content_hash`、`enabled`
- 入库任务：开始/结束时间、成功数、失败数、错误摘要
- 向量引用：`collection_name`、`point_count`、版本号（可选）

建议表（示例）：

- `kb_dataset`
- `kb_ingest_job`
- `kb_ingest_job_item`

#### 2）用户

建议存储：

- 用户主数据：账号、角色、状态、最后登录时间
- 会话主数据：`session_id`、用户 ID、创建时间、会话状态
- 对话消息：用户问题、模型回答、时间戳、消息类型
- 反馈：点赞/点踩、问题标注、人工修正建议

建议表（示例）：

- `user`
- `chat_session`
- `chat_message`
- `chat_feedback`

#### 3）日志

建议存储：

- 检索日志：query、命中条数、TopK、耗时、最高分
- 生成日志：模型名、token 使用、首 token 时延、总耗时
- 链路追踪：一次问答关联的 `session_id/request_id`、命中文档引用（point id）
- 异常日志：错误码、错误堆栈摘要、发生模块

建议表（示例）：

- `retrieval_log`
- `generation_log`
- `chat_trace`
- `error_log`

### 6.3 与向量库的边界

- MySQL 存“可管理、可统计、可审计”的业务数据。
- Qdrant 存“可检索”的知识向量数据。
- 两者通过 `source_file/content_hash/point_id` 建立引用，不重复保存正文。

## 九、实践方案

### 9.1 迭代原则

1. 先补主链路短板，不急着重做架构。
2. 优先复用现有模块，尤其是 `app/routers/chat.py`、`app/knowledge_base/retrieval`、`app/knowledge_base/ingestion` 和现有评测脚本。
3. 短期不建议把主链路重构成 LangChain Agent。当前仓库已经有适配层，但现有 LLM 封装并不适合直接承接原生 tool calling，继续走“显式检索 + 显式路由”更稳。

### 9.2 三阶段执行顺序

#### 第一阶段：把现有能力补到稳定可交付（1 到 2 周）

重点动作：

- 给聊天接口增加缓存命中和缓存失效策略。
- 把知识库入库任务从内存状态迁到数据库表，增加独立 worker 或轮询执行器。
- 为检索、重排、LLM 调用补 request_id、耗时日志和错误码。
- 给重排链路补降级逻辑，避免模型输出异常时整条链路失败。
- 建立最小可用自动化测试：认证接口、聊天接口、意图分类、检索过滤。

截至 **2026-04-22**，第一阶段规划内的核心项已经补齐：聊天缓存、统一错误结构、聊天与检索结构化日志、重排降级记录、入库任务状态持久化、后台轮询 worker、运行中任务重启恢复、失败任务人工重试、密码强度校验、认证日志、基础限流、会话续期接口、JWT 密钥轮换、登录失败临时锁定、关键接口审计日志、后端回归单测，以及认证/聊天/知识库任务接口测试与前端 Playwright 烟雾回归。后续重点不再是补第一阶段缺口，而是**推进 worker 独立进程化**并正式进入**第二阶段的多轮质量与答案结构化增强**。

阶段目标：当前演示环境在重启、超时、异常响应下仍能稳定运行；第一阶段剩余开发项已补齐。

#### 第二阶段：提高问答质量和多轮稳定性（2 到 3 周）

重点动作：

- 为多轮问答增加上下文摘要、槽位记忆和澄清追问。
- 把表单、保全、分公司、在售产品这些高频场景单独整理成回归测试集。
- 输出结构化答案对象，至少包含回答正文、引用证据、推荐动作和风险提示。
- 建立低分拒答和人工兜底策略，减少“看起来回答了，实际没答上”的情况。

截至 **2026-04-22**，第二阶段已经完成首个纵向切片：`/chat` 非流式接口已返回 `structured_answer`，`/chat/stream` 已补 `structured` SSE 事件，前端聊天页可展示依据摘要、建议下一步、风险提示和置信度；对应的后端单测与接口测试也已补齐。下一步应优先推进**多轮上下文管理**和**低置信兜底策略**，再扩高频业务场景回归集。

阶段目标：核心业务问题的回答稳定性和可解释性明显提升。

#### 第三阶段：补生产化能力（2 周起）

重点动作：

- 增加限流、审计日志、刷新令牌、配置分层和敏感信息管理。
- 建立性能基准脚本，固定输出检索耗时、LLM 耗时和端到端耗时。
- 制定部署、备份、回滚和异常恢复流程。
- 视业务目标再决定是否继续做推荐、线索转化和人工坐席联动。

阶段目标：项目从“可演示”进入“可维护、可上线试运行”的状态。

### 9.3 按模块落地建议

| 模块           | 建议做法                                                               | 原因                                           |
| -------------- | ---------------------------------------------------------------------- | ---------------------------------------------- |
| 聊天主链路     | 继续沿用当前 router + retrieval 结构，先补缓存、日志、异常和测试       | 现有链路已经能跑，直接推倒重来收益不高。       |
| 入库任务       | 使用现有 `KnowledgeIngestJob` 表做状态持久化，再配一个独立 worker 进程 | 这条路最贴近当前代码，也最容易落地。           |
| 检索质量       | 继续扩充 `evaluate_retrieval_quality.py` 的样本集，用固定问题集做回归  | 先把质量看见，再做优化，避免盲调阈值。         |
| 多轮问答       | 在现有历史窗口基础上补一层 Context Manager，而不是直接塞更多历史消息   | 单纯扩大窗口只会抬高成本，不会稳定提升质量。   |
| 前端           | 优先补错误态、空态、任务状态刷新和结构化结果渲染                       | 当前页面主流程已通，短板在细节反馈和可解释性。 |
| LangChain 集成 | 先保留适配层，不作为短期主线                                           | 当前更适合做兼容和实验，不适合承接核心流程。   |

## 十、结论

当前项目的判断不应再停留在“是否把 RAG 跑起来”。这部分已经完成。下一阶段真正要解决的是三件事：

1. 把现有主链路补到稳定可交付。
2. 把质量评估从手工体验改成固定数据集回归。
3. 把运维、安全、任务持久化这些生产问题提前补上。

如果按本文档推进，下一步最该优先完成的是 **第二阶段的多轮上下文管理和低置信兜底策略**，并并行推进当前轮询 worker 的独立进程化与高频业务场景回归集。这样既能延续第一阶段已经补齐的稳定性成果，也能把已经落地的结构化答案输出继续打磨成稳定可解释的业务能力。
